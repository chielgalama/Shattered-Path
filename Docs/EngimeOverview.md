# Shattered Path - Engine Overview

Version: 1.0

---

# Purpose

This document provides a high-level overview of the Shattered Path engine.

It is intended to explain how the major systems interact.

This document should remain stable as the project grows.

Implementation details belong in the source code, not here.

---

# Engine Philosophy

The engine is divided into four layers:

Definitions
        │
        ▼
Builders
        │
        ▼
Runtime
        │
        ▼
Gameplay Systems
```

Each layer has a single responsibility.

---

# Layer 1 - Definitions

Definitions contain only game data.

Definitions are editable by designers.

Definitions never contain gameplay logic.

Examples:

```text
CharacterDefinition
BaseItemDefinition
SkillDefinition
EnemyDefinition

AffixDefinition
AffixPoolDefinition

ProjectileDefinition

BuffDefinition
```

Definitions are implemented as ScriptableObjects.

---

# Layer 2 - Builders

Builders convert data into runtime objects.

Builders perform no gameplay.

Examples:

```text
CharacterBuilder

ItemBuilder

SkillBuilder

EnemyBuilder

ProjectileBuilder
```

Relationship:

```text
Definition
      │
      ▼
Builder
      │
      ▼
Runtime Object
```

---

# Layer 3 - Runtime Objects

Runtime objects represent active gameplay.

They contain mutable state.

Examples:

```text
Character

Enemy

ItemInstance

Projectile

Buff

Skill
```

Runtime objects own their own state.

They do not know how they were created.

---

# Layer 4 - Gameplay Systems

Gameplay systems coordinate runtime objects.

Examples:

```text
CombatSystem

LootSystem

AISystem

MovementSystem

StatSystem

SkillSystem

ProjectileSystem
```

Systems execute gameplay.

Runtime objects contain data and behaviour.

Systems coordinate interactions.

---

# Stats

The stat engine is shared by every gameplay object.

```text
             BaseStats
                 │
                 ▼
          StatBuilder
                 │
                 ▼
          StatCollection
                 │
                 ▼
               Stat
                 │
                 ▼
         ModifierCollection
```

The stat pipeline is identical for every object.

Examples:

Character

Enemy

Item

Projectile

Skill

Totem

Mine

Trap

Future gameplay systems should reuse the existing stat engine.

---

# Character

```text
CharacterDefinition
        │
        ▼
CharacterBuilder
        │
        ▼
Character
        │
        ├──────── Equipment
        │
        ├──────── Skills
        │
        ├──────── Buffs
        │
        └──────── Stats
```

Characters represent all living actors.

Examples:

Player

Enemy

NPC

Summons

---

# Equipment

Equipment stores equipped ItemInstances.

```text
Equipment

Helmet

Body Armour

Gloves

Boots

Weapon

Off-Hand

Rings

Amulet

Belt
```

Equipment itself contains very little logic.

Its responsibility is to manage equipped items.

---

# Items

Items exist in two forms.

## Definition

```text
BaseItemDefinition
```

Contains:

Base Stats

Implicit

Affix Pools

Tags

Requirements

Visual Data

---

## Runtime

```text
ItemInstance
```

Contains:

Local Stats

Generated Affixes

Sockets

Quality

Corruption

Future Runtime State

Items are generated by ItemBuilder.

---

# Skills

```text
SkillDefinition
        │
        ▼
SkillBuilder
        │
        ▼
Skill
```

Skills contain gameplay behaviour.

Skills reuse the stat system.

---

# Combat

Combat is coordinated by CombatSystem.

Combat never calculates stats itself.

Instead:

```text
Character

↓

Stat System

↓

Combat
```

Combat only requests final values.

---

# Loot

```text
Enemy

↓

LootSystem

↓

ItemBuilder

↓

ItemInstance

↓

Drop
```

Loot generation is independent of combat.

---

# AI

```text
Enemy

↓

AISystem

↓

Behaviour

↓

Actions
```

AI never directly modifies stats.

AI requests actions.

Actions use gameplay systems.

---

# Modifier Flow

All gameplay mechanics eventually become modifiers.

Examples:

Passive Tree

Items

Buffs

Debuffs

Auras

Skills

Map Modifiers

Future League Mechanics

Everything eventually contributes to the same modifier pipeline.

---

# Stat Flow

```text
Definitions

↓

Base Stats

↓

Runtime Stats

↓

Local Modifiers

↓

Global Modifiers

↓

Final Values

↓

Gameplay
```

Every gameplay system consumes final stat values.

No gameplay system should implement its own calculations.

---

# Dependency Direction

Dependencies always point downward.

```text
Definitions

↓

Builders

↓

Runtime

↓

Gameplay Systems
```

Lower layers never depend on higher layers.

Definitions never depend on Runtime.

Runtime never depends on Gameplay Systems.

# Engine Growth Strategy

New gameplay should extend existing systems whenever possible.

Preferred:

```text
New Feature

↓

Reuse Existing Systems

↓

Small Extension
```

Avoid:

```text
New Feature

↓

New Architecture

↓

Duplicate Logic
```

---

# Current Vertical Slice

The current development goal is:

```text
Player

↓

Equipment

↓

Stats

↓

Enemy

↓

Combat

↓

Death
```

Once this loop is complete, additional systems can safely be added.

---

# Long-Term Goal

The engine should remain modular.

New content should primarily be created through Definitions.

New gameplay should primarily be implemented through Gameplay Systems.

Runtime objects should remain lightweight.

The architecture should make it easy to add new content without rewriting existing systems.

---

# Guiding Principle

Every new feature should answer one question before implementation:

**Can this be built by extending the existing engine?**

If the answer is yes, extend the engine.

If the answer is no, identify the limitation first before introducing new architecture.
